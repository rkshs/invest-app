# Безопасность: техника и UX

Ключевые кейсы безопасности клиентского приложения основанные на OWASP MASVS (Mobile Application Security Verification Standard).
Ссылка на доку: https://mas.owasp.org/MASVS/

Целевой уровень для финансового приложения: **MASVS-L2 + R** (защита от реверса и тамперинга).

Статусы: `[ ]` — не сделано, `[~]` — частично / только demo, `[x]` — сделано.

---

## 1. Хранение данных (MASVS-STORAGE)

### Технические кейсы

- `[ ]` Access/refresh-токены хранятся только в `expo-secure-store` (Keychain / Android Keystore). Запрещено: `AsyncStorage`, MMKV без шифрования, обычные файлы.
- `[ ]` Пароль пользователя никогда не сохраняется на устройстве — только передаётся на сервер.
- `[ ]` PIN-код не хранится в открытом виде: либо проверка на сервере, либо локально хранится только хеш с солью + device-bound ключ из SecureStore.
- `[ ]` Ключ, разблокируемый биометрией, создаётся через `SecureStore` с `requireAuthentication: true` — привязка к биометрии на уровне ОС, а не флажок в коде.
- `[~]` Демо-стор (`src/features/auth/api/demo/demoAuthStore.ts`) хранит пароли и пины открытым текстом в `Map` — допустимо только для демо, не должно попасть в прод-сборку.

### UX-кейсы

- `[ ]` Экраны с балансом, портфелем и PIN скрываются в переключателе приложений (privacy screen: blur/заглушка при `AppState !== 'active'`).
- `[ ]` Запрет скриншотов на чувствительных экранах (PIN, реквизиты) через `expo-screen-capture`.
- `[ ]` Кнопка «Скрыть баланс» (глазок) на главном экране — пользователь сам управляет видимостью сумм в публичных местах.

## 2. Криптография (MASVS-CRYPTO)

- `[ ]` Все токены (access, verification, login) выпускает сервер криптографически стойко (JWT с коротким TTL или opaque-токены).
- `[ ]` Случайность на клиенте: только `expo-crypto` (`getRandomValues`), никогда `Math.random()`.
- `[ ]` Никакой самописной криптографии, только платформенные примитивы через Keystore/Keychain.

## 3. Аутентификация и сессии (MASVS-AUTH)

### Технические кейсы

- `[ ]` Реализация `apiAuthService` с парой access + refresh токенов, ротацией refresh-токена и серверной инвалидацией при logout.
- `[ ]` Серверный rate-limiting и счётчик попыток для OTP, пароля и PIN. Клиентские таймеры (`useResendTimer`).
- `[ ]` OTP: ограниченный срок жизни кода и лимит переотправок на сервере.
- `[ ]` Срок жизни сессии: expiry токена + авто-logout по бездействию.
- `[ ]` При logout очищается всё состояние: токены из SecureStore, кэш React Query (`queryClient.clear()`).
- `[~]` Биометрия: веб-ветка в `biometricAuth.ts` всегда возвращает успех(сейчас) не должна попасть в прод-логику.
- `[ ]` Для критичных операций (сделки, вывод средств) — `disableDeviceFallback: true`, чтобы нельзя было откатиться на пасскод устройства.

### UX-кейсы

- `[ ]` После N неверных PIN - понятный сценарий отката на вход по паролю.
- `[x]` Таймер переотправки OTP с обратным отсчётом (`useResendTimer`). Пользователь видит, когда можно запросить код снова.
- `[ ]` Понятные тексты ошибок без утечки деталей: «Неверный логин или пароль» вместо «Пользователь не найден» (не раскрывать, существует ли аккаунт).

## 4. Сеть (MASVS-NETWORK)

- `[ ]` Только HTTPS / TLS 1.2+. Для Android — `usesCleartextTraffic: false` (через `expo-build-properties`), для iOS — не ослаблять ATS.
- `[ ]` Certificate pinning для API-домена (TrustKit / OkHttp CertificatePinner через config plugin) с механизмом ротации пинов.
- `[ ]` Чувствительные данные не передаются в query-string только body/headers. (IOS и так по умолчанию не пропустит)
- `[ ]` UX: при ошибке пиннинга (возможный MITM) — отдельный экран «Небезопасное соединение», а не generic-ошибка сети.

## 5. Платформа (MASVS-PLATFORM)

### Технические кейсы

- `[ ]` Поля пароля и PIN: `secureTextEntry`, `autoComplete="off"`, `importantForAutofill="no"` защита от кэша клавиатуры и сторонних автозаполнений.
- `[ ]` OTP-автозаполнение через `textContentType="oneTimeCode"` (iOS) / SMS Retriever (Android), без чтения буфера обмена.
- `[ ]` Если появится "скопировать реквизиты", то нужно будет очищать clipboard по таймеру.
- `[ ]` Deep links / universal links: валидация всех входящих параметров, никаких переходов на чувствительные экраны без активной сессии.
- `[ ]` Заменить `com.anonymous.investapp` в `app.json` на реальный package/bundle id до подписания.

### UX-кейсы

- `[ ]` OTP подставляется в одно касание из SMS/пуша. Пользователь не переключается между приложениями.
- `[ ]` При открытии по deep link без сессии — сначала аутентификация, потом переход к целевому экрану.

## 6. Качество кода и зависимостей (MASVS-CODE)

- `[ ]` `npm audit` / Dependabot / Renovate в CI.
- `[ ]` Статический анализ: Semgrep (правила для React Native), ESLint security-плагины.
- `[ ]` Разделение demo/prod по env (EAS build profiles): `AUTH_API_MODE === 'demo'`, весь `src/features/auth/api/demo/` и моки (`mockClients`, `mockChats`, `mockLinkedDevices`) не попадают в релизную сборку.

## 7. Защита от реверса и тамперинга (MASVS-RESILIENCE)

- `[x]` Hermes - байткод вместо читаемого JS (по умолчанию в Expo).
- `[ ]` ProGuard/R8 + `shrinkResources` для Android (через `expo-build-properties`).
- `[ ]` Детект root/jailbreak (например, `jail-monkey`): предупреждение на входе, блокировка для операций со сделками.
- `[ ]` Детект отладчика и эмулятора в прод-сборке.
- `[ ]` Google Play Integrity API / Apple App Attest подтверждение целостности приложения на бэкенде.
- `[ ]` Никаких секретов (API-ключей) в JS-бандле, всё секретное живёт на сервере.
- `[ ]` UX: на рутованном устройстве, честное объяснение ограничений ("Операции недоступны, ваше устройство небезопасно"). Подумать над текстом ошибки.

## 8. Приватность и процесс (MASVS-PRIVACY)

- `[ ]` Threat modeling при подключении реального API: что хранится, что передаётся, модель атакующего.
- `[ ]` Прогон чек-листа OWASP MASTG перед релизами; для L2 — внешний пентест.
- `[~]` Моки с «реалистичными» клиентами (`src/entities/client/mockClients.ts`) не содержат реальных PII — проверять при каждом обновлении моков.
- `[ ]` Экран «Устройства» (`LinkedDevicesList`): пользователь видит активные сессии и может завершить любую.
