# Invest App — Архитектура

Документ описывает бизнес-контекст, ключевые архитектурные решения (ADR), стек, дизайн-систему и процессы разработки. Проект находится на стадии прототипа: бэкенда нет, данные замоканы, цель: миграция на реальный API (заложен в архитектуру, см. ADR-0003).

---

## 1. Бизнес-контекст

### Цели

Прототип мобильного инвестиционного приложения для демонстрации.

Роадмап:

1. Прототип экранов приложения - **текущий этап**
2. MVP (подключение бэкенда)
3. Бета-тестирование
4. Production-релиз

### Пользователи

Приложение обслуживает две роли с разными наборами экранов:

| Роль | Сценарии |
|---|---|
| **Клиент (инвестор)** | Портфель и счета в разных валютах, рынки, детали активов, чат с трейдером, настройки (PIN, пароль, устройства) |
| **Трейдер** | Список чатов с клиентами, саб-чаты, просмотр портфеля клиента, ответы на поручения, настройки |

Демо-учётка трейдера — в `README.md`.

### Основной процесс

Клиент авторизуется (логин/пароль → OTP → PIN → биометрия), видит портфель и рынки, отправляет поручения трейдеру через чат. Трейдер видит обращения клиентов, их портфели и отвечает в чате.

## 2. Архитектура проекта (ADR)

Формат: контекст → решение → последствия. Один ADR — одно решение.

### ADR-0001. Стек: Expo (SDK 56) + React Native + TypeScript

**Статус:** принято

**Контекст.** Нужен кроссплатформенный прототип с быстрой итерацией по дизайну и минимальными затратами на нативную инфраструктуру.

**Решение.** Expo SDK 56, React Native 0.85, TypeScript. Функциональные компоненты и хуки.

**Последствия.**
- (+) Быстрый старт, OTA-обновления, единая кодовая база iOS/Android/Web.
- (+) Готовые модули (биометрия, blur, градиенты).

### ADR-0002. Feature-based архитектура (FSD-подобная)

**Статус:** принято

**Контекст.** Прототип будет развиваться в MVP; нужно, чтобы код не превратился в монолит из экранов, а бизнес-логика не оседала в UI-компонентах.

**Решение.** Слоистая структура в `src/`:

```
src/
  app/        — навигация, провайдеры (композиция приложения)
  features/   — auth, chat, portfolio, trader, settings, client
                (внутри: api / model / lib / ui)
  entities/   — client, session (доменные сущности)
  shared/     — theme, lib, api (переиспользуемый код без бизнес-логики)
```

Бизнес-логика живёт в `model`/`lib`, UI-компоненты остаются презентационными. Композиция вместо наследования.

**Последствия.**
- (+) Фичи изолированы, их можно развивать и удалять независимо.
- (−) Технический долг: «дослойные» папки `src/components`, `src/screens`, `src/types` остались вне фичевой структуры. При переходе к MVP их содержимое нужно разнести по `features/` и `shared/`.

### ADR-0003. Мок-бэкенд через переключаемые сервисы (demo/api)

**Статус:** принято — ключевое решение прототипа

**Контекст.** Бэкенда нет. Путь миграции на реальный API должен быть определён заранее, чтобы демо-код не пророс сквозь всё приложение.

**Решение.** Фичи с данными работают через интерфейс сервиса (`AuthService`, `ChatMessagingService`) и фабрику (`getAuthService`, `getChatMessagingService`). Реализации две: `demo/` (in-memory моки) и `api/` (заглушка под будущий бэкенд). Выбор — компайл-тайм константой в `config.ts` фичи (`AUTH_API_MODE`, `CHAT_MESSAGING_MODE`). Статические справочные данные (клиенты, счета, бумаги) лежат в `mock*`-модулях рядом с потребителями.

Миграция в комментариях `src/features/auth/config.ts` и `src/features/chat/config.ts`.

**Последствия.**
- (+) UI и модель не знают, откуда приходят данные; подключение бэкенда не потребует переписывания фич.
- (−) Данные не переживают перезапуск; нет сетевых ошибок, латентности и пагинации — поведение с реальным API будет отличаться. Осознанный риск прототипа.

### ADR-0004. Состояние: React Context + TanStack Query, без Redux/Zustand

**Статус:** принято

**Контекст.** Глобального состояния немного: сессия, флоу авторизации, чаты трейдера. Полноценный стейт-менеджер пока избыточен.

**Решение.** Сессионное и флоу-состояние — React Context (`AuthContext`, `AuthFlowContext`, `TraderChatsContext`). Асинхронные операции и серверные данные — TanStack React Query (`useAuthMutations`, единый `queryClient` в `shared/api`).

**Последствия.**
- (+) Минимум зависимостей; React Query = кэш, ретраи и мутации, в будущем упростит подключение бэкенда.

### ADR-0005. Навигация: React Navigation с разделением по ролям

**Статус:** принято

**Контекст.** Две роли: клиент и трейдер с разными наборами экранов.

**Решение.** React Navigation (native-stack + bottom-tabs). Три корневых навигатора: `AuthNavigator`, `ClientNavigator`, `TraderNavigator`; корень выбирает навигатор по состоянию сессии и роли.

**Последствия.**
- (+) Роли полностью изолированы, нельзя случайно попасть в чужой стек.

### ADR-0006. Дизайн-система: StyleSheet + токены темы

**Статус:** принято

**Контекст.** Нужен консистентный вид по дизайн-референсам (`design/`) без подключения UI-библиотек.

**Решение.** Стили только через `StyleSheet.create()`, токены — в `shared/theme`.

**Последствия.**
- (+) Единый вид, смена палитры/шрифта в одном месте, нет тяжёлых UI-зависимостей.
- (−) Нет тёмной темы и рантайм-темизации — при необходимости понадобится ThemeProvider (отдельный ADR).

## 4. Дизайн-система и качество кода

### Стили

- Только `StyleSheet.create()`, инлайн-стили запрещены.
- Токены в `src/shared/theme`: `colors.ts`, `spacing.ts` (8pt-сетка), `typography.ts`.
- Шрифт Onest (400/500/600) подключается глобально через `applyDefaultFontFamily` в `AppProviders`.
- Консистентные радиусы скругления через токены.

### Компоненты

- Переиспользуемые компоненты: одна папка — один компонент (`ComponentName/index.tsx`).
- Компоненты презентационные: бизнес-логика в `model`/`lib` соответствующей фичи или в `shared/lib` (форматирование валют, дат, портфельные расчёты).
- Именованные экспорты, функциональные компоненты, хуки.

### Качество кода

- TypeScript без `any`; типы доменных объектов — в `src/types` и в `types.ts` фичей.
- Утилиты — маленькие чистые функции (`shared/lib`), легко покрываются тестами.
- Тестов пока нет, при переходе к MVP добавить Jest + React Native Testing Library.
- Линтер/форматтер (ESLint + Prettier) — добавить перед командной разработкой MVP.

## 6. Ветки

### Текущие

- `main` — основная ветка, всегда рабочее демо.
- `vercel-deploy` — web-деплой на Vercel.

### Соглашение (GitHub Flow) в будущем

- `main` защищена, изменения — только через PR.
- Рабочие ветки от `main`: `feature/<название>`, `fix/<название>`.
- PR небольшие, один PR — одна задача; после мержа ветка удаляется.
